## 2026-06-17 - TASK-031 (T26): Display filters

**What was done:** Implemented M9/T26 — `--quiet` and `--show` display filters (PRD §3.8):

- `src/output/human.ts`: Extended `HumanRenderOptions` with `quiet?: boolean` and `show?: Severity`.
  Added `SEVERITY_RANK` map (error=0, warning=1, info=2). `--quiet` skips phases with
  `status === "completed"` and `findings.length === 0`. `--show <severity>` filters per-finding
  display using the rank: only findings where `SEVERITY_RANK[finding.severity] <= showRank` are
  shown; if all filtered, the phase shows "no findings". Cost footer and result line always
  rendered. Exit code is never touched by display filters.

- `src/output/human.test.ts`: 11 new T26 tests — `--quiet` suppresses passing phases but not
  phases with findings or non-completed phases; result/cost lines survive all-suppressed case;
  mixed phases (one passing, one with findings) — only passing suppressed. `--show error` hides
  warning+info; `--show warning` shows error+warning, hides info; `--show info` shows all;
  `--show error` with no matching findings shows "no findings"; exit code unchanged when show
  filter hides the gating finding from display.

- `src/cli.ts`: Added `quiet: boolean` and `show?: Severity` to `ParsedFlags`. Added `quiet`
  and `show` options to `parseArgs`. Added `--show` validation (same `VALID_FAIL_ON` vocab as
  `--fail-on`). Wired both into the `renderHuman` call at step 9.

**Learnings:**
- Severity ordering is error > warning > info. Model as a rank map (error=0, warning=1, info=2)
  and show finding if `SEVERITY_RANK[finding.severity] <= showRank`. This is identical to the
  gating filter pattern in exit-codes.ts.
- `--quiet` suppresses at the phase level using the *original* findings count (before `--show`
  filtering), so a phase with only info findings is still "passing" and suppressed by `--quiet`
  even if `--show error` would also hide those findings.
- `--show` and `--quiet` are purely display concerns: exit code derivation uses the original
  RunReport (unchanged), so `--show error` can hide the gating warning finding from display
  while the exit code remains 1.
- The `vp check --fix` reformatter flattened some of the test fixture calls; always run it
  before committing.

## 2026-06-16 - TASK-029 (T24): Large-diff visibility + semantic pre-filtering

**What was done:** Implemented M8/T24 — semantic diff pre-filter + budget enforcement (PRD §3.6):

- `src/diff-filter.ts`: New module. `filterDiff(files, diff) → FilterResult`. Strips lockfiles
  (package-lock.json, yarn.lock, pnpm-lock.yaml, bun.lockb, Cargo.lock, go.sum, etc.),
  minified files (*.min.js, *.min.css), sourcemaps (*.map), vendored directories (vendor/,
  vendors/, third_party/, external/, extern/, node_modules/), and `@generated`-annotated
  diff sections — EXCEPT migration files (paths containing "migrat"). Stripped paths returned in
  `strippedFiles` (sorted), non-stripped diff text in `filteredDiff`. Files without diff sections
  still get path-based filtering. 27 unit tests.

- `src/phases/coverage.ts`: New module. `applyBudget(diff, budget, phaseId) → BudgetResult`.
  Splits diff into per-file sections, greedily includes sections in file order until budget is
  reached, returns truncated diff + excluded paths + `<phase>.partial-coverage` warning Finding
  (severity:"warning", confidence:"high"). No truncation when diff fits budget. Exports
  `DIFF_BUDGET = 200_000`. 13 unit tests. Warning is a valid Finding (Value.Check passes).

- `src/scheduler.ts`: Added `diff?: string` to `SchedulerContext`; forwarded to `PhaseContext`
  in `runPhaseGuarded` after applying `applyBudget` per-phase. If truncation occurred, the
  partial-coverage warning is prepended to the phase's findings (before report merge).

- `src/cli.ts`: Added `getDiffText(cwd, scope)` helper — runs appropriate git command for scope
  kind (staged: `git diff --cached`, working: `git diff HEAD`, against: `git diff ref...HEAD`,
  commit: `git diff sha^1 sha`; commits: empty string, range not recoverable from Scope).
  Degrades to empty string on git failure. Added `filterDiff` call between scope detection and
  runPhases; when files are stripped, the scope handed to phases carries `files: filteredFiles`
  (post-filter list) plus `stripped: strippedFiles`, so `files ∪ stripped` = original and the
  risk classifier (composite.ts → classify(diff, scope.files, rules)) consumes pre-filtered paths
  too, not just the pre-filtered diff (PRD #32). Scope echo reports the post-filter file count
  plus the stripped count; passes filteredDiff to SchedulerContext.diff.

**Learnings:**
- The `commits` scope kind doesn't store the original range string in the Scope object (only
  files are stored). `getDiffText` can't recover it → returns "" and filters by path only.
  If `--commits` is used, @generated stripping requires diff content and is skipped.
- `filterDiff` produces an empty `filteredDiff` when the diff input is empty (no @generated
  stripping) — path-based rules still apply since they operate on `files`, not diff content.
- The per-file diff section parser splits on `/(?=^diff --git )/m` — a positive lookahead that
  preserves the delimiter at the start of each section. This is the same pattern used in both
  diff-filter.ts and coverage.ts (kept local to avoid coupling).
- `vp check --fix` reformatted regex patterns (e.g. `/^vendor\//` → `path.startsWith("vendor/")`)
  and removed redundant `/.../` → `.endsWith(...)` — the formatter prefers string methods over
  simple prefix/suffix regexes.
