## 2026-06-11 - T12: Phase-level budgets (wall-clock + turns)

**What was done:** Implemented T12 — phase-level budget enforcement for wall-clock and turn-count limits.

- **`src/agent/budgets.ts`** (new): exports `runWithWallClock(runner, inputs, controller)` — races the runner promise against a `wallClockMs` timeout. On timeout, calls `controller.abort()` and returns `Err(BudgetError{ limit: "wallClockMs" })`. On early completion, clears the timeout timer and returns the runner result unchanged.
- **`src/agent/fake-runner.ts`** (extended): added `DelayScript { kind: "delay", delayMs }` to the `RunScript` union. The delay fake respects the abort signal — when the wall-clock controller aborts, the fake resolves immediately with `Err(CancelledError)` (clean teardown; the race already fired with `BudgetError` so CancelledError is discarded).
- **`src/phases/agent-phase.ts`** (wired): replaced the direct `runner.run()` call with `runWithWallClock()`. A fresh `AbortController` is created per run invocation.
- **`src/agent/budgets.test.ts`** (new): 20 tests covering wall-clock race (via `vi.useFakeTimers()` + `vi.advanceTimersByTimeAsync`) and turn-count (scripted `Err(BudgetError)` through `makeAgentPhase`). Both paths verified to produce `PhaseReport{ status: "error", reason: "budget exceeded", audit: {} }`.

**Layering enforced (plan §2a/P10):** wrapper owns wall clock; runner owns turns + bash limits. Turn-count tests use `FakeAgentRunner` scripted to return `Err(BudgetError{ limit: "turns" })` — the fake doesn't re-implement enforcement, just exercises the wrapper's `matchError` path.

**Learnings:**
- `vp` isn't in PATH in this environment — use `node_modules/.bin/vp test` instead. After `bun install --cwd /workspace/repo` installs deps, the binary works.
- `vi.useFakeTimers()` + `vi.advanceTimersByTimeAsync(ms)` is the clean approach for testing timer-based races in Vitest — no real sleeps needed. Call `vi.useRealTimers()` in `afterEach` to avoid test pollution.
- The `DelayScript` fake needs to use `signal.addEventListener("abort", ...)` to resolve promptly when the wall-clock controller aborts. Without this, the fake's `setTimeout` is still pending after the race fires, causing timer leaks in tests (vitest would hang waiting for them).
- `PhaseId` pattern `^[a-z][a-z0-9-]*$` rejects dots — don't use dotted finding ids as phase ids in tests. Finding ids like `"x.test"` are fine (they're `string`, not `PhaseId`).
- vp check --fix reformatted `budgets.test.ts` and `fake-runner.ts` (trailing whitespace + line wrapping). Always run `vp check --fix` before committing.

## 2026-06-11 - T13: Bash-level limits (timeout + output cap)

**What was done:** Implemented T13 — bash-level limit enforcement (timeout + output cap).

- **`src/agent/budgets.ts`** (extended): added `BASH_TRUNCATION_MARKER`, `RunBashOptions`, `RunBashResult`, and `runBash(command, options)`. The function uses `child_process.spawn` with `detached: true` (so the entire process group can be killed via `process.kill(-pid, "SIGKILL")`). On timeout, kills the process group and returns `{ timedOut: true, output: <output so far> }`. On output cap exceeded, truncates and appends `BASH_TRUNCATION_MARKER`, kills the process group, returns `{ truncated: true }`.
- **`src/agent/pi-runner.ts`** (extended): imported `createBashToolDefinition`, `BashOperations`, `ToolDefinition`, and `runBash`. When "bash" is in the toolset, filters it out of the string toolset and creates a custom bash tool definition using `createBashToolDefinition` with a `BashOperations.exec` that calls `runBash`. The custom bash tool replaces the built-in bash with our limit-enforcing version.
- **`src/agent/budgets.test.ts`** (extended): 14 new tests covering timeout (sleep + emit-before-hang), output cap (printf, yes), normal completion (echo, exit codes, stderr), and the exact marker constant. All tests use real `child_process.spawn` (no fake timers), with small timeoutMs (50ms) and outputCap values.

**Architecture note (plan §2a/P10):** Bash limits are transparent to the phase — the agent gets output-so-far + the truncation marker, not a BudgetError phase failure. This is distinct from T12's wall-clock limit (which produces Err(BudgetError)).

**Learnings:**
- With `shell: true` spawn, killing the shell process does NOT kill its children (e.g. `yes` in `sh -c yes`). The children inherit the shell's stdout pipe and keep it open indefinitely, causing test hangs. Fix: spawn with `detached: true` so the child gets a new process group (pgid = child.pid), then kill the group with `process.kill(-child.pid, "SIGKILL")`.
- `createBashToolDefinition` returns `ToolDefinition<TObject<...>, BashToolDetails | undefined, BashRenderState>` which is not directly assignable to `ToolDefinition` (the base type with defaults). TypeScript requires an explicit `as ToolDefinition` cast when adding to `customTools: ToolDefinition[]`.
- `createBashToolDefinition` and `createLocalBashOperations` are exported from `@earendil-works/pi-coding-agent` main index — no need for deep imports.
- The `BashOperations.exec` callback-based API (`onData: (data: Buffer) => void`) works fine with our `runBash` return value by calling `onData(Buffer.from(result.output))` once with the full accumulated output after `runBash` returns.
- `vp check --fix` reformatted the `import type` block in `pi-runner.ts` from inline to multi-line.
