## 2026-06-12 - TASK-023: T18 · Malformed & unknown-key handling
**What was done:** Implemented T18 — unknown top-level config keys now produce `harness.unknown-config-key` warning findings (severity: warning, confidence: high) instead of silently passing through or erroring. Changed `loadConfig`'s return type from `Result<StetConfig, ConfigError>` to `Result<{ config: StetConfig; findings: Finding[] }, ConfigError>`. Unknown key detection happens per-layer in `readYamlLayer` (so the finding names the file it came from). Two files (user and project) with unknown keys each emit their own finding. The CLI now destructures `{ config, findings: configFindings }` and injects a synthetic `phase: "harness"` PhaseReport into the assembleReport call when configFindings are non-empty. Malformed YAML → Err(ConfigError) was already implemented in T17. Updated all 13 pre-existing load.test.ts precedence tests to use `result.value.config.*` instead of `result.value.*`, and added 7 new T18 describe block tests.
**Learnings:**
- Changing `loadConfig`'s return type is the clean approach — it keeps the unknown-key warning co-located with the config data and forces callers to handle both.
- `KNOWN_TOP_LEVEL_KEYS = new Set(Object.keys(StetConfig.properties))` is the TypeBox-idiomatic way to get schema property names; this is a Set of ["phases", "output"].
- The synthetic "harness" PhaseReport needs phase: "harness" (valid PhaseId), status: "completed", findings: [...], audit: {}, cost: { durationMs: 0 }. "harness" matches `^[a-z][a-z0-9-]*$`.
- `vp check --fix` auto-formats; remember to remove unused `Finding` import after restructuring.
- bun install modifies package.json with workspace catalog fields — must revert with `git checkout -- package.json` (T17 recurring trap).
