## 2026-06-16 - TASK-028 (T23): Spec-context combining

**What was done:** Implemented M8/T23 — spec-context combining (PRD §3.6):

- `src/spec-context.ts`: New module. `buildSpecContext(input, deps)` accepts `--prd <file|-|literal>`
  and `--task <string>`, tries to read prd as a file (ENOENT → treat as literal), concatenates
  prd text + task text with `\n\n`, and returns `SpecContext { text, sources }` where sources is
  the CLI-form list (e.g. `["--prd docs/spec.md", "--task"]`). Injectable `readFile` and
  `readStdin` deps keep the module fully testable. Returns `Err(ConfigError)` for real fs errors.

- `src/spec-context.test.ts`: 13 tests covering all paths: no flags, literal prd, file prd,
  stdin prd (`-`), task only, prd+task concatenation, stdin+task, EACCES error, stdin error,
  sources derivation contract.

- `src/phases/types.ts`: Added `spec?: string` to `PhaseContext` — phases that consume spec
  context read this field.

- `src/scheduler.ts`: Added `spec?: string` to `SchedulerContext`; forwarded to `PhaseContext`
  in `runPhaseGuarded`.

- `src/report.ts`: Added `spec?: { provided, sources }` to `AssembleInput`; `assembleReport`
  uses `input.spec ?? { provided: false, sources: [] }` instead of the hardcoded M8 placeholder.

- `src/cli.ts`: Added `--prd` and `--task` to `parseArgs` options and `ParsedFlags`; imported
  and called `buildSpecContext` between flag parsing and scope detection (step 3b); wired spec
  text into `runPhases` and spec sources into `assembleReport`. Updated help text.

- `src/report.test.ts`: Updated "Slice 5" test + added spec passthrough test.

**Learnings:**
- TypeScript does not narrow `Record<string, string>[key]` to `string` after a `key in obj`
  check — it remains `string | undefined`. Use `const v = obj[key]; if (v !== undefined)` instead.
- `vp check --fix` fixes formatting but does NOT fix type errors — run `vp check` after `--fix`
  to catch remaining issues.
- `isFileAbsentError` from `src/fs-util.ts` is the right tool for ENOENT detection — it covers
  both ENOENT and ENOTDIR, which is the established project pattern.
- The `SpecContextDeps` injection pattern (separate `deps` param with defaults) is cleaner than
  adding fields to `CliIo` — the CLI uses real fs by default, tests inject mocks directly.

## 2026-06-14 - TASK-035 (T29): Deterministic risk classifier + level→fan-out/coordinator wiring

**What was done:** Implemented M7.5 steps 5–6 — the deterministic risk classifier
mechanism (PRD §3.4.1a, decisions #26, #32):

- `src/risk/classify.ts`: Pure function `classify(diff, paths, rules) → level`.
  Rules are `RiskRule[]` (ordered predicate/level pairs); first match wins; falls back to
  "full" (conservative) if no rule matches. 16 table tests cover all edge cases.

- `src/risk/classify.test.ts`: Table tests for `classify` across fixture rules, path-sensitive
  rules, rule evaluation order, empty/no-match fallback, and multi-level rule sets.

- `src/phases/types.ts`: Added `diff?: string` to `PhaseContext`. Absent until M8 lands
  (pre-filter will populate it); composite.ts passes `ctx.diff ?? ""` to classify.

- `src/schema/report.ts`: Added `level?: string` to `PhaseReport` TypeBox schema. The
  resolved level appears in the run output per acceptance #19. Optional field — fully
  backwards-compatible, existing tests unaffected.

- `src/phases/composite.ts`: Added `RiskLevel` interface and `riskRules?`/`riskLevels?`
  to `CompositePhaseConfig`. Before the specialist fan-out, if `riskRules` is declared,
  `classify()` is called and the level drives: (a) filtering which specialists run
  (`levelSpecialists` Set), (b) whether the coordinator is skipped
  (`skipCoordinatorForLevel`). With no `riskRules` the mechanism is inert. `levelEntry`
  is spread into every completed/error `PhaseReport` when a level was resolved.

- `src/phases/stub-composite.ts`: Exported `STUB_RISK_RULES` (lines > 10 ⇒ "full" else
  "trivial") and `STUB_RISK_LEVELS` ({ trivial: { specialists: ["alpha"], coordinator: false },
  full: { specialists: [...all], coordinator: true } }). `makeStubComposite` opts extended
  to accept `riskRules` and `riskLevels`.

- `src/phases/stub-composite.test.ts`: 9 new T29 wiring tests — small diff → level
  "trivial" → only alpha runs, no coordinator; large diff → level "full" → all three +
  coordinator runs and audit.coordinator is populated; inert mechanism (no riskRules →
  all three run regardless); schema compliance with level field; PhaseReport.level round-trip.

**Learnings:**
- The `levelEntry` const (for spreading level into PhaseReport) must be defined BEFORE the
  `Promise.all` try/catch, so it's accessible in the catch path (unexpected error return).
  Defining it after the catch block meant it was unavailable in the error branch.
- `bun install` is available in this environment (PATH: /home/agent/.bun/bin); `pnpm`/`npx`
  are not. Use `node_modules/.bin/vp` directly after `bun install`.
- `vp check --fix` formats both `package.json` and test files; always run it before committing.
- The `package.json` gets a `workspaces.catalog` + `overrides` block added by `bun install`
  (bun lockfile migration artifact). This is harmless under the project's pnpm/vp toolchain
  but was flagged as a blocker in the T27 review (commit 28272d9). The formatter does not
  revert it, so it remains in `package.json` post-install. Noting for future agents: this is
  a known bun-install artifact, not a product change.
- TypeBox `additionalProperties: false` schemas require the new field to be added to the
  schema object for `Value.Check` to pass — just adding to the TS return type is not enough.
