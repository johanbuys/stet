01What stet is

An independent code-change validation suite: a standalone CLI that judges a change across five dimensions and reports structured findings. Invoked when work is believed done — by a human pre-commit, by an AI agent at the end of a task, by an autonomous loop, or by CI. stet reports; it never fixes. There is no --fix.

02Why — the killer demonstration

The R&D proved the gap no static layer covers. Two byte-identical sibling repos differ by one line: the broken one prints the right error but is missing process.exit(1) — bad input exits 0.

✅ Correct sibling

  • ✓ tests green
  • ✓ static review passes
  • ✓ behavioral run passes
$ node cli.js bad.json; echo $?
Error: invalid JSON
1

❌ Broken sibling (missing process.exit(1))

  • ✓ tests green — asserts the message, never the exit code
  • ✓ static review passes — the try/catch is "there"
  • ✗ behavioral run fails — with the reproducing command
$ node cli.js bad.json; echo $?
Error: invalid JSON
0   ← violates the acceptance criterion
Only running the system catches it. An author-agent grading its own work compounds this — the POC documented agents over-claiming repeatedly. The validator must be independent of the author, and verification that never executes inherits whatever the tests and the author missed.

03Design principles

P1Findings, not fixes
No write tools, at any phase, by construction. A verdict from a mutated repo is worthless. Remediation belongs to the caller.
P2Independent verification
stet runs the gates itself; behavioral claims derive from the spec, never the diff. The diff only selects which surfaces to exercise.
P3Evidence must reach the claim
Code shape ≠ proof. A mock proves the harness, not the product. A pipe is not a TTY. HTTP is not proof for an SPA.
P4Blunt conservative rubrics
A precise permissive rule is a loophole for a weaker model. Proven by a measured rollback in the R&D.
P5Verdicts inside, findings outside
Phase 5 reaches a real verdict with executable evidence, then surfaces it as findings through the same severity contract as everything else.
P6Parallel by default
Phases are independent; sequencing is a cost-policy knob, not architecture. Fail-fast cancellation bounds wasted tokens.
P7Agentic at the edges, deterministic in the loop
LLM exploration is spent once at stet init; the recurring Phase 1 run path is plain process execution.
P8Inference over configuration
Zero-config works; init upgrades it; config refines it; flags override.
P9Model routing is config
Per-phase model knob. Robust models for anything touching real systems/mocks/web; cheap models routed off the trap classes.
P10Provisioned, never self-installed
The browser is provisioned ahead (baked image / remote / cloud) — never installed mid-validation.
P11The eval suite is the discipline
14 fixtures + content-aware grader gate every rubric edit and model-routing change, under vp test.

06Phase 5 up close — one rubric × execution adapters

The single most reusable R&D finding: the judgment is surface-agnostic; only the execution adapter is surface-specific. Nothing in the rubric says "CLI."

JUDGMENT — one hardened rubric, reused

  • Diff-blind: claims derive from the spec; diff only selects surfaces
  • Mutation-free: no write tools, enforced by construction
  • Evidence sufficiency: evidence must reach the regime the claim is about
  • Anti-mocking: a mock proves your harness, not the product
  • Interactive honesty: a pipe is not a TTY; unreachable ≠ failed
  • Browser rule: SPAs need a real browser; curl is not proof

EXECUTION ADAPTERS — per surface

  • CLI  spawn → stdout / stderr / exit code
  • API  start_service → curl → status / body
  • WEB  start_service → agent-browser → DOM / network
  • TTY  pty_session → raw-mode keys
Tools earned by probes, not speculation — built only where models could not self-serve reliably.

07CLI contract

Scope (auto-detected, flags override)

stetstaged → working tree → branch vs default → last commit
--staged --workingexplicit scope for scripted contexts
--against <ref> --commit --commits

Spec context

--prd <file|-|text>file, stdin, or inline literal
--task "<text>"inline task description
--issue <n>delegates to gh — no built-in forge integrations; other platforms pipe to --prd -

Exit codes — one knob

codemeaning
0clean at threshold
1findings at/above --fail-on (default error)
2tool error (no scope, context fetch failed…)

Default: only a proven failed blocks. Strict CI opts into --fail-on warning to also gate on blocked / skipped-behavioral. AI findings below high confidence never gate. Output: human-readable default, --format json for machines, SARIF later. Headless by design — no TTY, container-friendly.

08stet init — the exploration agent

The one place agentic exploration is spent so runs never spend it. The same exploration that finds vp test also finds how to start the app — so init drafts Phase 5's run-instructions too, turning the hardest config in the product into "review what init drafted."

Explore (read-only)

09Out of scope

--fix / any remediation (deliberate cut from v1) visual regression perf benchmarking license / dependency scanning multi-language beyond TS/JS IDE plugins watch mode server / daemon mode plugin system distributed execution

Cutting --fix also deletes fix-convergence loops, fix-scoping rules, and "don't fix the tests" policing — and makes the whole product mutation-free by construction, not just Phase 5.

12Things to poke at while reviewing

Questions the draft answers implicitly or leaves for feature PRDs — worth a deliberate look.

Is "full suite" still right at the margin?
Phases 2–4 overlap with what code-review agents increasingly do. Phase 5 is the moat. Does the roadmap invest accordingly — and would you cut a static phase before cutting anything in Phase 5?
Cancellation semantics
Gates fail → cancel AI phases. But should a lint failure cancel as aggressively as a compile failure? Per-gate cancellation policy might belong in the harness feature PRD.
Confidence × severity gating
"AI findings below high confidence never gate" — is that the right line for spec-compliance errors (a stated requirement unmet is fairly binary)? Phase 5 findings are evidence-backed; should they bypass confidence entirely?
Large diffs
Context-window overflow strategy (chunking vs. degrade-with-warning) is unaddressed — v1 left it open too. Harness concern.
Cost visibility
Loops care about $/run. Should the report include per-phase token/cost accounting as a first-class output field?
Concurrent behavioral runs
start_service implies ports/databases. Two stet runs on one CI box: isolation story (port allocation, teardown guarantees) — feature-PRD depth, but worth flagging now.
The spec-quality dependency
Phases 2 and 5 are only as good as the spec given. Garbage spec → confident-looking findings about the wrong thing. Should stet judge spec adequacy ("this spec has no testable acceptance criteria") as a finding?
