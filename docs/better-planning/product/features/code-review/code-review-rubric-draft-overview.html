code-review — specialist rubrics

A visual overview of code-review-rubric-draft.md. The concrete starting rubrics for stet's Phase 3, to be settled against the eval suite — not hand-perfected.

The shape, in one line: specialists report broadly under a hard concrete-scenario bar → a coordinator verifies against real code and may only drop noise, never evidence-backed findings → severity gating stays downstream. This is the field-validated finder → verify → judge pattern, which stet already implements.

Specialists run in parallel, each in its own context (one subagent per check). Recall is tuned in the finders; precision in the coordinator.

Shared preamble — the 6 pillars (prepended to every specialist)

🔒 Independent & read-only

Doesn't trust the author's claims; judges the change itself. No write tools by design — reports, never fixes.

🎯 Scope to the introduced change

Review only what this diff introduces (net vs base). Pre-existing code is flagged only if re-exposed, and tagged meta.preexisting.

🧪 Concrete-scenario bar strongest rule

Flag only what you can explain with a specific input/state → wrong outcome. Can't construct it? Don't flag it. Prefer not reporting over guessing.

👁 Partial-context honesty

The diff is often budget-trimmed. Don't flag "undefined / missing import / incomplete" for symbols that may live elsewhere — read first. Don't claim breakage without the specific call site.

🤫 Abstention is a good result

An empty list is valid. Cap at {MAX_FINDINGS}. Fewer, higher-confidence findings beat an exhaustive list.

📏 Conventions

Respect repo CLAUDE.md. To flag a violation, quote the exact rule and the exact line — no "spirit of the doc."

The four specialists

Correctness / bugs can emit error

review.bug.*

  • Inverted conditions, off-by-one, null deref, missing await, falsy-zero, swallowed errors, races.
  • Edge cases: empty input, root commit, over-budget, odd formats; regex anchors/backtracking.
  • Reads the enclosing function — bugs in unchanged lines of a touched function are in scope.

Security can emit error

review.security.*

  • Injection, git/CLI option injection, path traversal, prototype pollution, secret leakage, SSRF.
  • Key precedents (don't flag): framework escaping on by default; env vars & CLI flags are trusted; test/markdown ≠ prod vuln.
  • Requires an attacker-input → impact path; states reachability and sets confidence to match.

Quality / maintainability warning max

review.quality.*

  • Re-implementing an existing helper (must name it), redundant state, copy-paste, dead code.
  • Wrong-altitude abstractions; wasted work on hot path/startup; scope-retaining closures.
  • States the concrete cost + the simpler form. No taste/naming opinions.

Coverage-gaps warning max

review.coverage-gap.*

  • New/changed branches the diff's tests don't exercise — error paths, edges, boundaries.
  • Judges whether tests would actually fail if the code were wrong, or just mirror it (tautology/mock-only).
  • Only genuine gaps; doesn't demand tests for trivial code.

The coordinator — verify & judge

Constrained authority

  • Verifies each candidate against the real code before keeping it — drops stale / unreproducible findings (the fix for the pressure-test's stale-finding problem).
  • Drops: duplicates, speculative-without-a-path, pure style, convention-contradicted.
  • MUST NOT drop or downgrade evidence-backed or deterministic findings — reinstates them unchanged if it tries.
  • Every drop & reinstatement is recorded in audit.coordinator (received / dropped / reinstated).
  • Ranks survivors; prefers confidence from specialist agreement over a single self-reported number.

Open calibration questions — settle against eval, not now

  1. Confidence model — self-reported (current) vs agreement-based (2-of-3 verify voters). Research says self-reported is miscalibrated. The one fork I'd want your call on.
  2. Finding caps & risk levels — {MAX_FINDINGS} per specialist; how the risk classifier scales the panel and toggles the coordinator.
  3. Pre-existing handling — a meta.preexisting flag vs a separate tier vs suppression.
  4. Few-shot examples — add 3–5 good/bad findings per specialist (research's top lever); draft from eval fixtures once the harness exists.
  5. Specialist set — is performance separable from quality? a dedicated "patterns" specialist worth it?
  6. Severity ↔ gating — confirm the gate stays downstream and no specialist is muzzled.
