03Agreement-based verify — the confidence model

The headline decision. Self-reported confidence is miscalibrated; we derive it from independent corroboration instead.

1
Specialists roll up candidate findings (report broadly — not muzzled).
2
Each candidate is sent to N=3 independent voters, each told to refute it from a distinct lens.
perspective-diverse beats N identical refuters
3
Confidence = agreement: 3/3 → high · 2/3 → medium · ≤1/3 → dropped
4
Coordinator dedups, drops convention-contradicted/speculative, applies the pre-existing tier, re-ranks. Cannot drop evidence-backed/deterministic findings — reinstates if it tries; logs all of it to audit.coordinator.

04Risk dial → how much to spend

Deterministic; once per run, before fan-out. Config-overridable and eval-tuned. Security is never dialed away on sensitive paths.

trivial
small diff, no sensitive paths → bugs only, coordinator off
standard
→ bugs + quality + coverage-gaps, coordinator on
full
large or sensitive (auth/crypto/migrations) → all four + security always, coordinator on

08Resolved in review round 1

All six open items were resolved in the 2026-06-19 canvas round (recommendations accepted). Kept below as the record; the resolutions are in §07 and the markdown decisions table.

Is N-voter agreement-verify a harness amendment, and do we do it now or stub it?
The current coordinator is a single judge pass. Agreement-confidence (decision 1, 12) needs N independent voters. Either we amend harness-prd first, or v1 ships with the single-judge coordinator and "agreement" is approximated by the judge — which weakens the headline decision. Which?
Is 4 specialists right, and is folding perf+patterns into quality a false economy?
Field norm is ~one prompt per defect family. Folding keeps cost down but the eval may show quality misses architecture/perf issues. Start folded and let eval decide, or split now?
N=3 voters per candidate is 3× verify cost — acceptable, or scale N by severity/risk?
e.g. bugs/security get 3 voters, quality/coverage get 1. Cheaper, but confidence then means different things per specialist. Uniform vs tiered?
Is the review-coverage-gaps vs Phase-4 test-quality split clean enough to avoid double-reporting?
Review judges "is this code tested?"; Phase 4 judges "are these tests meaningful?". On a diff that adds both code and weak tests, both could fire. Need a sharper rule or an explicit handoff?
v1 eval fixtures: hand-author, or stand up the CR-Bench-style SWE-bench mining pipeline?
Hand-authored is faster to start but small and biased; mining real PRs is richer but is its own build. Start hand-authored + a few mined, full pipeline in the eval-suite feature?
How does a specialist reliably decide a finding is pre-existing vs introduced?
meta.preexisting is settled as a tier, but the *detection* (is this line in the diff's additions, or only in surrounding context?) needs a concrete rule. Diff-line membership, or a judgment call?
