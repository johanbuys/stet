The Harness
The shared substrate all five phases run on. A phase is reduced to a configuration; the harness owns everything common. Every other feature PRD writes against the contracts on this page.
STATUS: DRAFT — updated 2026-06-06 REVIEW ROUND 1 RESOLVED — all 6 open questions contracts mined from validation-agent-poc detail: docs/better-planning/product/features/harness/harness-prd.md

01Division of labor — harness owns, phases contribute

The boundary rule: needed by two or more phases → harness; needed by one → that phase's PRD. Adding a sixth phase must touch no harness code.

THE HARNESS OWNS

scope detection config layers + precedence scheduler + cancellation agent runner (Pi SDK) tier routing + qualification check findings schema output-as-tool + guards specialist fan-out + roll-up report aggregation budgets & presets cost accounting human / JSON output exit codes + signal handling teardown

A PHASE CONTRIBUTES ONLY

  • id · kind — deterministic | agent
  • activation — predicate over (diff, spec, config)
  • rubric — constant system prompt (cacheable)
  • toolset — allowlist; never edit/write
  • model — tier (robust | fast) or pin
  • extension — submit-schema additions
  • budgets — overrides of harness defaults
  • specialists — optional parallel specialists (§04)
Mutation-free by construction: the agent session's tool list is the phase allowlist — edit/write tools are never registered, so no phase (or specialist) has a code path to modify the repo. Verified by a test asserting the registered tool list.

03Output-as-tool — the agent completion contract

Each agent run — phase or specialist — has exactly one way to finish: a submit_findings tool whose parameter schema is the findings schema + the phase's extension. Three guards, each earned by an observed failure in evals.

1
Agent works — read / bash / grep / find / ls (+ Phase 5 execution tools). Freetext reasoning in, no parsing anywhere.
2
Agent calls submit_findings — the only path to completion (D6). Loose freetext in, rigorous JSON out (D7).
3
Schema-validate-or-retryGUARD 1
Input validated at the tool boundary; invalid → rejected → model retries. The schema is enforced, not hoped for.
4
IdempotencyGUARD 2
First successful submission wins; duplicates get "already recorded — stop now." (Observed: some models submit 10–13×.)
5
No-submit fallbackGUARD 3
Agent ends without submitting → harness synthesizes an error report + <phase>.no-result warning finding. A phase can never silently vanish.

04Specialists — composite agent phases NEW — review round 1

A panel of narrow specialists beats one generalist with a kitchen-sink rubric. A phase may declare specialists: parallel sub-agents, each the same configuration shape (rubric + toolset + model + activation), findings rolling up to the parent. Phase = reporting unit; specialist = execution unit. All three §03 guards apply per specialist; one specialist failing never loses the others' findings.

Question-3 resolution lives here: Phase 4 (test quality) activates only when the diff touches test files — it judges the tests as artifacts (tautology, mirroring, would-it-fail). "Tests missing or stale" is a judgment about the code → the coverage-gaps specialist, which rides the always-on review phase for zero extra activations. Specialists inherit the phase tier unless overridden (security on robust, quality on fast is plausible). Fences: custom user-defined specialists deferred (plugin system); no cross-specialist dedup pass in v1.

05Model routing — tiers, not IDs NEW — review round 1

User A has OpenAI, user B has Anthropic, user C has OpenRouter — same repo. Absolute model defaults can't work, and provider pins in shared config are broken-by-teammate. So defaults are capability tiers, resolved at run time against whichever providers this user has credentials for.

phase wants tier: robust→ shipped per-provider preference table× credentialed providers (Pi SDK registry + auth)→ best available qualified model→ report records cost.model

Four config layers — resolved per-setting

1 · flags
this invocation — --model behavioral=… (bare --model X = all agent phases; specific beats general) · --budget thorough · --only/--skip
2 · project config
stet.config.yml, checked in — project facts: gate commands, run-instructions, tier intent. Never names providers → safe for teams. stet init writes ONLY this layer, sparsely.
3 · user config
~/.config/stet — machine/provider facts: "robust means X for me", local model qualifications. Never project-specific.
4 · built-in defaults
shipped — tier table with receipts (= the curated qualification manifest), budget defaults, scheduler policy. Sparse configs track these as they improve.

Qualification — tier membership is earned on the eval suite

mechanismcontract
stet models test <provider/id>runs the fixture suite N times (default 3); verdict per tier; any false passed on an over-claim trap disqualifies; conservative misses tolerated. User pays their own model's exam.
Qualification key(model × rubric version × fixture-set version) — a rubric edit or new trap invalidates prior quals; stale quals downgrade to the warning, never silent acceptance.
Registrycurated manifest shipped per release + local qual cache in the user layer + community PRs. No web service, no self-reported scores.
Unqualified model in useharness.unqualified-model WARNING finding — in the report, visible to loops; strict CI refuses unqualified judges via --fail-on warning; default runs anyway.

Whose result counts? — variance is contained, not eliminated

AI findings are judgments — even one model disagrees with itself across runs. Containment: everything that gates by default is model-independent (deterministic gates; Phase 5 faileds carry a reproducing command anyone can run); the conservative rubric makes cross-model disagreement skew toward extra warnings, not false passes (the grader's asymmetry); and the binding run is CI's run, with pinned routing — local runs are advisory pre-flight. cost.model makes any divergence diagnosable at first look. No consensus machinery in v1.

06Scheduler — activation, cancellation classes, teardown

Parallel default (wall-clock ≈ slowest phase) · sequential in config for staged cost · --continue-on-failure disables cancellation. Non-activated phases appear as skipped with the rule named.

phaseactivates when
gatesalways (unless --skip gates / config)
spec compliancespec context present
code reviewdiff non-empty — per-specialist predicates may narrow (coverage-gaps: non-test code added/changed)
test qualitydiff touches test files only — the tests themselves are the object of judgment
behavioraldiff touches runnable surfaces and spec present — surfaces changed without a spec ⇒ behavioral.not-run warning

The gate signal — two cancellation classes

CANCEL CLASS — "the code doesn't function"

teststypesbuild

Failure cancels in-flight AI phases. Their reports: cancelled, reason "gates failed: <gate>". Bounded token waste.

REPORT-ONLY CLASS — "the code is untidy"

lintformat

Failure becomes a finding but cancels nothing. Overridable per gate: gates.<name>.cancel. A gate timeout is always report-only — a merely-slow suite must not nuke the AI phases; only a failing gate proves the code doesn't function.

Phase outcome states

completed
ran to the end and submitted — may contain findings
skipped
activation rule / config / --skip — reason named; mandated skips also emit a finding
cancelled
scheduler cancelled it (gate failure, interrupt) — never silent
error
the phase itself broke — model unavailable, budget exhausted, no-submit

Teardown is total: sessions disposed, child process groups killed, Phase 5 services torn down. Graceful interrupt still writes the partial report — a partial run produces a partial report, never nothing. Second Ctrl-C force-kills (no report — teardown was refused).

07Budgets & safety — limits are the default

The POC ran with no limits and produced 300s+ hangs in evals. Every breach is a named outcome, never a silent hang or silent kill. Numbers adjusted in review round 1: the POC's successful web runs took ~8 min — a false "budget exceeded" costs a loop a whole iteration and is worse than a slow pass.

limitdefaulton breach
per-phase wall clock5 min (static agents) · 15 min (gates, behavioral)phase error, reason "budget exceeded", partial audit preserved; gate timeout never cancels other phases
per-phase turns50same
bash command timeout60 s — deliberate pressure toward start_service for anything long-livedcommand killed; output-so-far returned to the agent so it can react
bash output cap32 KB per calltruncated, with a visible marker

All per-phase overridable, and bundled: --budget <fast|default|thorough> (config budgets.preset) — fast for snappy pre-commit, thorough for loops that prize completeness over latency.

09Decisions — original draft + review round 1

decisioncall
Severity vocabulary● One gating vocab (error/warning/info); POC granularity survives in meta.
Audit trail● First-class for every phase — the anti-silent-green mechanism.
Confidence × evidence● Evidence-backed + deterministic findings are high-confidence by construction.
Per-gate cancellation● tests/types/build cancel; lint/format report-only; timeouts always report-only.
Budget numbers + presets● 15 min behavioral/gates, 5 min static agents; --budget fast|default|thorough.
CLI cleanup● --skip/--only generalized; --severity→--show; --model repeatable + namespaced.
Model routing● Capability tiers resolved against credentialed providers; 4 config layers; sparse init; CI is the binding run.
Model qualification● stet models test + curated manifest + local cache; harness.unqualified-model warning.
Specialists● First-class composite phases; review = 5-specialist panel; Phase 4 stays a phase, tests-only activation.
Signal exit codes● SIGINT→130, SIGTERM→143, partial report written; 2 strictly tool error.
Streaming● Deferred — run-then-read confirmed; NDJSON addable later without touching RunReport.
SDK● Confirmed: badlogic/pi-mono → earendil-works/pi (repo transferred); old npm scope deprecated in favor of @earendil-works/pi-coding-agent 0.78.x — the POC already runs on it.
